# Code of Conduct

## Purpose

Skills-Lab is a public lab for Desk Suites: ChatGPT skill systems that guide professional workflows, package domain procedures, and hand constrained work to CLI and coding agents. The project is open to maintainers, contributors, users, researchers, and reviewers who work in good faith to improve the quality, safety, and usefulness of those systems.

This code of conduct sets the minimum behavior standard for all project spaces, including GitHub issues, pull requests, discussions, release comments, documentation, examples, and related project communication.

## Expected behavior

Contributors are expected to:

- Treat other participants with professional respect.
- Keep critique technical, specific, and actionable.
- Assume good faith until evidence says otherwise.
- Use clear reproduction steps when reporting defects.
- Preserve project taxonomy, naming conventions, release structure, and skill packaging standards.
- Disclose relevant constraints, uncertainty, generated content, and source limitations.
- Prefer source-grounded corrections over unsupported claims.
- Respect maintainers' decisions on scope, release timing, taxonomy, and quality gates.

## Unacceptable behavior

The following behavior is not acceptable in project spaces:

- Harassment, threats, intimidation, stalking, or sustained disruption.
- Personal attacks, insults, or inflammatory comments unrelated to the work.
- Discriminatory language or conduct.
- Publishing private information without explicit permission.
- Submitting spam, malicious links, credential material, malware, or intentionally unsafe instructions.
- Misrepresenting generated content, external research, licensing, release integrity, or source provenance.
- Pressuring maintainers to bypass validation, checksum, packaging, security, or review gates.

## Technical conduct expectations

Skills-Lab is a repo of operational guidance systems. Contributions should be held to a higher standard than casual prompt collections. Contributors should avoid broad, ambiguous, or unverifiable changes. A change should state what it affects, why it is needed, how it preserves existing conventions, and how it was validated.

Do not submit secrets, production credentials, private user data, proprietary source text without authorization, or copyrighted material that the project cannot redistribute. Do not submit destructive automation examples unless they are explicitly framed as prohibited or safely isolated test fixtures.

## Enforcement

Maintainers may edit, hide, reject, or remove contributions that violate this policy. Maintainers may also restrict participation when behavior creates unnecessary risk for users, contributors, or the project.

Enforcement is based on project impact, not intent alone. A contributor may be asked to revise language, narrow scope, add citations, provide validation evidence, or stop participating in a thread.

## Reporting conduct issues

Report conduct concerns through GitHub by opening a minimal issue that requests maintainer contact, without including private details. For security-sensitive matters, use the process in `SECURITY.md` instead.

Maintainers should handle conduct reports with discretion and only share details with people required to review or resolve the issue.
