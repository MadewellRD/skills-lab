# PRD template

Use this structure for `product-requirements-document.md`.

## How to use this file

Use this PRD as the product source of truth for technical discovery, architecture, issue planning, testing, verification, and implementation handoff.

## Summary

One short paragraph describing the product change.

## Problem

Describe the user/customer/business problem. Separate evidence from interpretation.

## Goals

- Goal 1
- Goal 2

## Non-goals

- Explicitly out-of-scope item

## Users and stakeholders

- User or stakeholder
- Need or decision context

## Requirements

Use stable IDs.

| ID | Requirement | Source | Priority | Status |
|---|---|---|---|---|
| PRD-001 | Requirement statement | source fact or assumption | must/should/could | proposed |

## Acceptance criteria

| Requirement ID | Acceptance criterion | Verification method |
|---|---|---|
| PRD-001 | Clear observable result | test, review, demo, metric, or manual check |

## Risks

- Product, technical, security, compliance, or delivery risk.

## Open questions

- Question, owner if known, and blocking status.

## Downstream handoff notes

Summarize what later SDLC desks need and what must not be assumed.

## Source facts

- Verified fact and source reference.

## Unverified assumptions

- Assumption and why it matters.
